# Report the connection path once and force paths through the serve-minted locator

Forced-relay proof work (#4) and the throughput benchmarks need to know whether a session travels direct or via relay, so the connection path became product surface: after `run` or `pull` connects, the client prints one stderr line naming the path selected at connect time (`dumbridge: connected directly` / `via relay` / `(path unknown)`), and the path crosses the `BridgeTransport` seam as the iroh-agnostic `ConnectionPath` value on every session. The line is deliberately a single connect-time snapshot rather than a live watcher: iroh can upgrade a relayed session to a direct path mid-connection, and an unreported upgrade is honest as long as the docs say so, while a streaming path UI would complicate stderr for a one-request CLI. The forcing flags live on `serve` (`--direct-only` / `--relay-only`), not on `run`/`pull`, because the dialing endpoint derives its `RelayMode` from the locator inside the key — a direct-only key carries no relay URL, so every client dial is automatically direct-only with `RelayMode.disabled()` and fails fast with the branded `BridgeDirectConnectError` when holepunching fails — which keeps path policy with the session-policy owner and needs no client-side coordination. `--relay-only` constrains only the initial dial and is documented as best effort, never a guarantee: a hard relay-only mode requires Rust `clear_ip_transports()`, which `@number0/iroh@1.0.0` does not expose, so a true forced-relay test mode waits on iroh-ffi patch work rather than being faked here.
