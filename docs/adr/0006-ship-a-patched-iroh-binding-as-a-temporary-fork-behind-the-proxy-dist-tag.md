# Ship a patched iroh binding as a temporary fork behind the proxy dist-tag

Proxy-jailed cloud sandboxes (Claude Code web, Codex Cloud) allow egress only through an HTTP forward proxy and block UDP, so the iroh relay WebSocket is the only viable connection path there — and the stock `@number0/iroh` binding cannot route it: iroh's Rust core has supported both required knobs for some time (`proxy_url` / `proxy_from_env` since n0-computer/iroh PR #2298, covering the WebSocket relay path since #3217, and additive extra CA roots via `CaTlsConfig::with_extra_roots` since #3973 / #4300), but the Node-API binding surfaces neither, up to and including `@number0/iroh@1.1.0`. There is no "just upgrade" escape, and dropping a patched `.node` in via `NAPI_RS_NATIVE_LIBRARY_PATH` is blocked by the sandboxes' own safety policies. So dumbridge publishes a patched build of the binding as `dumbridge-iroh@1.0.0-proxy.0`, consumed by the `dumbridge@0.3.1-proxy.0` prerelease through the npm alias `"@number0/iroh": "npm:dumbridge-iroh@1.0.0-proxy.0"`, quarantined on the `proxy` dist-tag while `latest` stays on stock. The fork invents no behavior: its ~50-line patch (`patches/iroh-ffi-proxy.patch`, manifest pinned in `patches/dumbridge-iroh.package.json`) only re-exposes configuration that already exists in the Rust crate, and the mechanism — HTTP CONNECT for the relay WebSocket plus additive CA roots for TLS-intercepting proxies — is the same idiom Tailscale's DERP proxy path and the Codex CLI's own CA handling use, and the abstraction level iroh's maintainers deliberately chose over pluggable dialers.

Consequences accepted with this decision. Relay-only reachability under a proxy is an environmental constraint, not a dumbridge policy: an HTTP CONNECT tunnel carries TCP only, so UDP/QUIC holepunching can never traverse it and there is no direct-path recovery to design for in a jail. The `latest` dist-tag must not point at the fork while it ships two native targets (darwin-arm64, linux-x64-gnu) against stock's eleven — the alias replaces the binding on every platform, so promoting it would regress Windows, musl, and arm installs; `docs/patches/iroh-ffi-proxy.md` records that release gate. The fork sits outside the repo's trusted-publishing pipeline and rides an unscoped npm name, and every upstream binding release costs a manual rebase, rebuild, and republish. We also inherit iroh issue #4111 (`net_report` ignores the builder's `proxy_url` during home-relay selection): dumbridge clients only dial out — the relay comes from the locator in the bridge key, not from home-relay selection — and the sandboxes export the proxy variables that the probes' HTTP client honors by default, but a proxied environment that ever needs to *listen*, or one configured with an explicit proxy URL and no environment variables, may fail home-relay mounting until upstream fixes it.

This decision retires by upstreaming, not by growing: the exit is a PR to `n0-computer/iroh-ffi` adding the same passthrough builder methods (`proxyUrl`, `proxyFromEnv`, `caExtraRootsPem` — no dumbridge policy in any of them). Once a stock `@number0/iroh` release surfaces them, dumbridge drops the alias and depends on stock again; dumbridge already feature-detects the methods rather than pinning to the fork, so the swap is a dependency line. The fork is acceptable only while that exit is being pursued.
