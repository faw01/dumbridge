# Extend the dumbridge/1 grammar in place for reject and banner frames

Branded key errors need a server-to-client rejection signal and the first-run banner needs the sanitized root display on the wire, so `dumbridge/1` gains a terminal `reject` frame (bounded codes `invalid-key`, `expired-key`, legal only in place of a whole response) and a `banner` frame (sanitized served-root display, first frame of the first run response). The grammar was extended in place rather than minting `dumbridge/2` because both bridge sides ship in one unpublished prerelease CLI, and the protocol's strict schema decoding already makes any mixed-version pairing fail closed with a deterministic decode error rather than a silent misread. The reject is sent only after the constant-time capability comparison, so unauthenticated callers learn nothing they could not already infer from the ALPN; the expired-key check consequently moved from before request parsing to immediately after authentication, which still keeps expired keys away from the shell and every pull. Once a version ships to real users, further grammar changes require the protocol version bump this decision deliberately skipped.
